This might be a well known issue, but I couldn't spot it in the FAQ or 
bug list or mailing list archives.

If you execute Sys.sleep(60) in R.app, the command can't be interrupted 
by ESC or Cmd-C until it is done.  R.app becomes completely unresponsive.

The number 60 isn't important, as far as I can tell this is true for any 
sleep time.

A simple workaround is to limit the duration of Sys.sleep() calls, e.g. 
using a function like this:

  Sleep <- function(duration) {
   seconds <- floor(duration)
   for (i in seq_len(seconds))
     Sys.sleep(1)
   Sys.sleep(duration - seconds)
  }

R.app handles interrupts every time Sys.sleep returns, so this means the 
biggest delay is 1 second.

With?R 4.6 >= r89829? we're seeing this install.packages() error on Mac 
sonoma-arm64 in various places:

 ? ? Error in av.src[pkg.bin, "Version"] : subscript out of bounds

This happens for example with install.packages("OSTA.data", 
repos=BiocManager::repositories()).

Also started to happen in this GitHub action a few days ago: 
https://github.com/HyrienLab/igblastr/actions/runs/24495733061/job/71590105822

The error occurs in the new utils:::.available.both() function 
introduced at r89827 in trunk and?r89829 in R-4-6-branch (file 
src/library/utils/R/packages.R). It can simply be reproduced with:

utils:::.available.both("https://cran.r-project.org")

 ? ? # Error in av.src[pkg.bin, "Version"] : subscript out of bounds

It seems that the function should be able to handle the situation where 
some binary packages in the repo don't have a corresponding source 
package. Something like this maybe:

.available.both <- function(repos, method, ...) {
 ? ? ## "both" gets complicated as we have to use available.packages twice
 ? ? fields <- c(tools:::.get_standard_repository_db_fields(), "File", 
"Built")
 ? ? av.src <- available.packages(contriburl = contrib.url(repos, 
"source"), method = method, fields = fields, ...)
 ? ? av.bin <- available.packages(contriburl = contrib.url(repos, 
"binary"), method = method, fields = fields, ...)
 ? ? ## we keep things simple in that we use source unless there is the 
same binary version
 ? ? ## so that we can get at the Built field. This is more simple than 
in install.packages
 ? ? ## since we only care about determining whether a package is new, 
but not which type is
 ? ? ## preferred - install.packages will sort that out.
 ? ? all_src_pkgs <- row.names(av.src)
 ? ? all_bin_pkgs <- row.names(av.bin)
 ? ? dual_pkgs <- intersect(all_src_pkgs, all_bin_pkgs)
 ? ? src_ver <- av.src[dual_pkgs, "Version"]
 ? ? bin_ver <- av.bin[dual_pkgs, "Version"]
 ? ? use_bin <- as.numeric_version(bin_ver) >= as.numeric_version(src_ver)
 ? ? ## If for some reason 'src_ver' or 'bin_ver' contain NAs, they 
would propagate
 ? ? ## to 'use_bin', but can this ever happen?
 ? ? use_bin[is.na(use_bin)] <- TRUE
 ? ? src_pkgs <- c(setdiff(all_src_pkgs, dual_pkgs), dual_pkgs[!use_bin])
 ? ? bin_pkgs <- c(setdiff(all_bin_pkgs, dual_pkgs), dual_pkgs[use_bin])
 ? ? ## Returned matrix has 1 row per package in 'union(all_src_pkgs, 
all_bin_pkgs)'.
 ? ? rbind(av.src[src_pkgs, ], av.bin[bin_pkgs, ])
}

Peter,

there are separate PACKAGES files for the source repository and each of 
the binary repositories.
A PACKAGES file should always be consistent with the contents in the 
corresponding repository, of course. It must not contain entries for 
packages that are unavailable in that repo.
Technically, in order to avoid race conditions in the source repo, the 
entry in PACKAGES is even removed even before the package (or at least a 
link to it) is moved  away from the source repository.

For binary repositories: Binaries are not instantly removed from binary 
repositories if corresponding sources are moved to the archives. So you 
cannot expect that the source repo is always a superset of packages of a 
binary repo.

"both" has been always based on the (bad) assumption that binaries are a subset of sources. Before the update, it has simply ignored binaries entirely when determining the set of packages (i.e. both old.packages and available.packages were always wrong for "both" as they simply ran on "source"). It causes a lot of problems which is why my preferred solution would be to remove "both" entirely as the concept is fundamentally broken (see PR#18396), but since the horse bolted we have to deal with it for now to make it at least seem like it's working. As you noted, In practice, it is close to impossible to maintain that assumption, because those are two entirely independent processes, so archiving etc. would have to be done in a different way by removing binaries first and waiting for it to propagate - which won't happen anytime soon.

repos <- "https://cran.r-project.org"
av.src <- available.packages(contrib.url(repos, "source"))
bin_types <- c(Windows="win.binary", Mac="mac.binary.sonoma-arm64")
master <- c(repos, "https://mac.r-project.org")
for (i in seq_along(bin_types)) {
    av.bin <- available.packages(contrib.url(master[i], bin_types[[i]]))
    bin_pkgs <- setdiff(rownames(av.bin), rownames(av.src))
    cat(names(bin_types)[[i]], "packages with no source:", length(bin_pkgs), "\n")
}
