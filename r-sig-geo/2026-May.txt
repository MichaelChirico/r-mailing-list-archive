Hello experts

I fail to provide a reproducible example but try to outline the
observed behavior as good as I can...

I should read spatial data from a PostgreSQL database. I manage to set
up a connection using the DBI and RPostgres packages:

con <- DBI::dbConnect(
      RPostgres::Postgres(),
      host = 'my.host.net',
      dbname = 'dbname',
      user = 'myusername,
      password = 'mypassword'
    )

The spatial data ('mylayer') that I am interested in is stored as a
table in a schema ('thatschema') on the database. I can download the
data with:

dat <- st_read(con, layer = DBI::Id("thatschema", "mylayer"))

I can also access the data with:

dat <- st_read(con, query = "select * from thatschema.mylayer")

Given this, I did expect that the following would also work:

dat <- st_read(con, layer = "thatschema.mylayer")

But it throws an error:

ERROR: Failed to prepare query : ERROR: relation "from
thatschema.mylayer" does not exist
LINE 1: SELECT * FROM "from thatschema.mylayer"

Inspecting the traceback and involved functions plus a short test on
my usecase suggest that this could be dealt with inside
sf:::st_read.DBIObject by adding a single line before calling
DBI::dbReadTable (line 48 in the function). Something like:

layer <- Id(unlist(strsplit(layer, "\\.")))

I'm not enough of an expert to judge if this causes unwanted effects
in other use-cases, but I think that sf::st_read would gain
user-friendliness if it allowed specification of layers in the form
"thatschema.mylayer" for PostgreSQL databases, as I would think I am
not alone with coming across spatial databases with schemata involved.

Feedback appreciated...

Best, Rafael

Rafael,
You should be able to give a GDAL connection string for PostgreSQL as
the dsn argument to st_read(), which has format:

dsn <- "PG:host='addr' port='5432' dbname='databasename' user='x' password='y'"

See https://gdal.org/en/stable/drivers/vector/pg.html

You can override the active schema ('public' being the default) by
also specifying 'active_schema=schema_name' within that connection
string. Then you should be able to use st_read(dsn, layer =
"mylayer").

Note that it is also possible to configure a schema search path on the
Postgres server (i.e., by adding "thatschema" in your case), in which
case the client should not have to specify it.

Chris
