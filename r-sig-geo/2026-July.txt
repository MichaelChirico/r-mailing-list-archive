I am fitting a Geographically Weighted Regression (GWR) using the *GWmodel*.
The model is fitted at a coarse spatial scale and used to predict at a
finer spatial scale. My data are in a projected coordinate system (metres).
The formula is simple: one response variable, one predictor.

Instead of the standard isotropic kernel, I want a kernel whose shape
reflects the geometry of a satellite sensor's point spread function (PSF).
Specifically:

- The sensor has a cross-track viewing angle *?* that varies spatially
across the scene. I have this as a raster at the same resolution as my
response variable.
- The PSF expands in the cross-track direction as 1/cos?(*?*). At nadir (*?*
= 0) the kernel is symmetric. As *?* increases toward the swath edge, the
kernel becomes increasingly elongated in the cross-track direction.
- The cross-track direction is not axis-aligned ? it runs roughly
bottom-left to top-right in my scene.

Rather than modifying GWmodel's internal distance function, I pre-compute a
custom distance matrix and pass it via the *dMat* argument, which GWmodel
supports in *bw.gwr*(), *gwr.basic*(), and *gwr.predict*(). The distance
between any two points is computed as follows:

1. Estimate the cross-track orientation angle *?* from the spatial gradient
of the viewing angle raster.
2. For each regression point *i*, rotate the coordinate frame by ?*?* so
that the x-axis aligns with the cross-track direction.
3. Scale the cross-track component of each distance vector by 1/cos?(*??*),
where *??* is the viewing angle at point *i* extracted from the raster.
4. Compute Euclidean distance in this rotated and scaled space.

This produces an n ? n distance matrix for fitting (calibration points vs
calibration points) and an m ? n matrix for prediction (prediction points
vs calibration points), both passed as *dMat*/*dMat1*/*dMat2* to the
relevant GWmodel functions.

Is injecting a pre-computed anisotropic distance matrix via dMat
mathematically equivalent to modifying the kernel function itself, and is
this therefore a valid way to implement a spatially-varying anisotropic
kernel in GWmodel?

# 1. Load built-in data
data(LondonHP)
proj4string(londonhp) <- CRS("+init=epsg:27700")
sp_fit     <- londonhp[, c("PURCHASE", "FLOORSZ")]
fit_coords <- coordinates(sp_fit)

# 2. Synthetic VA raster
# Values increase linearly BL->TR (30-35 deg). Alpha is known analytically:
# BL->TR diagonal = 45 deg from East, so alpha = pi/4.
bb  <- bbox(sp_fit)
buf <- 2000

va_ext <- ext(bb[1,1] - buf, bb[1,2] + buf,
              bb[2,1] - buf, bb[2,2] + buf)

va  <- rast(va_ext, resolution = 500, crs = "EPSG:27700")
xy  <- crds(va)
va_vals <- 30 + 5 * ((xy[,1] - bb[1,1]) / diff(bb[1,]) +
                       (xy[,2] - bb[2,1]) / diff(bb[2,])) / 2
values(va) <- va_vals

alpha <- pi / 4   # BL->TR diagonal, known analytically

# 3. Anisotropic distance matrix function
make_aniso_dmat <- function(data_coords, reg_coords, va_reg, alpha) {
  n_d   <- nrow(data_coords)
  n_r   <- nrow(reg_coords)
  D     <- matrix(NA_real_, nrow = n_r, ncol = n_d)
  cos_a <- cos(-alpha)
  sin_a <- sin(-alpha)
  for (i in seq_len(n_r)) {
    dx      <- data_coords[, 1] - reg_coords[i, 1]
    dy      <- data_coords[, 2] - reg_coords[i, 2]
    u       <-  cos_a * dx - sin_a * dy
    v       <-  sin_a * dx + cos_a * dy
    scale_i <- 1 / cos(va_reg[i] * pi / 180)^2
    D[i, ]  <- sqrt((u * scale_i)^2 + v^2)
  }
  D
}

# 4. Extract VA at calibration points
va_fit                <- extract(va, fit_coords)[, 1]
va_fit[is.na(va_fit)] <- mean(va_fit, na.rm = TRUE)

# 5. Calibration distance matrix
dMat_fit <- make_aniso_dmat(fit_coords, fit_coords, va_fit, alpha)

# 6. Bandwidth selection
bw <- bw.gwr(
  formula  = PURCHASE ~ FLOORSZ,
  data     = sp_fit,
  approach = "AIC",
  kernel   = "gaussian",
  adaptive = TRUE,
  dMat     = dMat_fit,
  parallel.method = "omp",
  parallel.arg = 4
)

# 7. Fit GWR
gwr_fit <- gwr.basic(
  formula  = PURCHASE ~ FLOORSZ,
  data     = sp_fit,
  bw       = bw,
  kernel   = "gaussian",
  adaptive = TRUE,
  dMat     = dMat_fit,
  parallel.method = "omp",
  parallel.arg = 4

print(gwr_fit)

Using a pre-computed distance matrix through the dMat argument is indeed a reasonable way to introduce anisotropic effects into a GWR model. However, I noticed that the Euclidean distance is finally calculated with coordinates rotated and scaled. It could be problemetirc to represent the actually anisotropic features in the spatially varying relationships.

I would suggest redefining the kernel function itself by defining the parameter kernel="kernelfun" (write a new kernel funciton namely kernelfun), or even you can replace the input weight matrix by modifying the function gwr.basic, e.g.:
 W.i<-gw.weight(dist.vi,bw,kernel,adaptive)
      W[,i] <- W.i

Geocomputation and Machine Learning for Environmental Applications
(intermediate level;  September, October, November, 2026)
(https://www.spatial-ecology.net/pages/courses/2026/open_call_fall2026)
On this course, students will be introduced to an array of powerful
open-source geocomputation tools and machine learning methodologies in
the Linux environment. Students who have never been exposed to
programming in Linux will acquire confidence in using advanced open
source data processing routines. Those with a programming background
will find the course beneficial in improving their programming and
modelling skills. We aim to equip attendees with powerful programming
tools, as well as hone their abilities for independent development.
This will be valuable not only for GIS related applications but also
for general data processing and applied statistical computing in a
number of fields. We strive to provide the best grounding for career
development as a geographic data scientist.

More information and registration: www.spatial-ecology.net
