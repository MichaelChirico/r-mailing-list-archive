 Assumed-size arrays in fortran and memory sanitizer
Message-ID: <AS8P192MB17023BF62C0E49FE2161A17193F62@AS8P192MB1702.EURP192.PROD.OUTLOOK.COM>

Hi all,

Thank you all so much for your useful answers!


I managed to solve the issue, by replacing the calls to xDCOPY that led to the overlapping memory by two new subroutines, XDMOVE (for vectors) and XDMOVE2D (for matrices).
(The explanation below is a bit technical.)


*The problem was:*

The original xDCOPY is defined as:
      SUBROUTINE XDCOPY(N, DX,INCX, DY,INCY)

where DX is the vector to copy *from* and DY is the vector to copy *to*, and INCX and INCY are the increments for DX and DY respectively in the DO loop.

An error could be triggered if xDCOPY was used, for instance, as:

           CALL xDCOPY (N, X(J+1), 1, X(J), 1)

as it was called with two overlapping subsections from the same vector (X)

*The solution*

For such cases, a new subroutine, called xDMOVE is now used,
that is defined as:

        SUBROUTINE XDMOVE(N, DX, Ifrom, INCfrom,   Ito, INCto)

where vector DX is passed only once, and
*Ifrom* and *Ito* are the *index* of the first position to copy *from* and to parts of DX.


For instance, the original call:
           CALL xDCOPY (N, X(J+1), 1, X(J), 1)
becomes:
           CALL xDMOVE (N, X, J+1,1,    J,1)

and:
          CALL xDCOPY(N, X,   0, X,  1)
becomes
          CALL xDMOVE(N, X, 1,0,   1,1)




When the input array is a *matrix*, xDCOPY is replaced by xDMOVE2D that is defined as follows:

        SUBROUTINE XDMOVE2D(N, MDX, DX, Ifrom,Jfrom,INCfrom, Ito,Jto,INCto)

where MDX is the leading dimension of the assumed-size matrix DX(MDX,*),
and Ifrom,Jfrom and Ito,Jto are the indices to the first position to copy from and to in DX(I,J).


In practice, XDMOVE2D calls XDMOVE, with the indices to the vector that corresponds to DX:

      ...
      IXvec = (Jfrom - 1)*MDX + Ifrom
      IYvec = (Jto       - 1)*MDX + Ito

      CALL XDMOVE(N, DX, IXvec,INCfrom, IYvec,INCto)



As an example of a call, the original call :
          CALL xDCOPY   (M-ME,    W(ME+1,J), 1, W(KRANKE+1,J), 1)
becomes:
          CALL xDMOVE2D (M-ME, MDW, W, ME+1,J,1, KRANKE+1,J,1)

where W is an assumed-size matrix defined as W(MDW, *)



So, it was not as difficult to fix as I originally feared!
Thanks again for all the valuable suggestions.

it tests (making it possible to test packages with dependencies not on
CRAN/BioC), but that shouldn't normally cause problems during incoming
CRAN checks like you're currently having. This looks like a crash some

There's no obvious reason for a crash or resource starvation.

A package that was checked soon after lme4 failed to install due to
'rlang' failing to load:
https://win-builder.r-project.org/incoming_pretest/rotations_1.6.7_20260705_010208/Windows/00install.out
Maybe this was Uwe reinstalling rlang to fix the problem?

Otherwise you might have to switch to a more verbose reporter for your
next CRAN submission in order to locate the place of the crash more
precisely:

LogReporter <- R6::R6Class('LogReporter',
 inherit = testthat::Reporter,
 public = list(
  file = '', failures = list(),
  initialize = function(...)
   super$initialize(...),
  start_file = function(filename)
   self$cat_tight("File: ", self$file <- filename, "\n"),
  start_context = function(context)
   self$cat_tight(" Context: ", context, "\n"),
  start_test = function(context, test)
   self$cat_tight("  Test: ", test),
  add_result = function(context, test, result)
   if (inherits(result, c('expectation_success', 'expectation_skip'))) {
    self$cat_tight("."); flush(stdout())
   } else {
    self$failures <- c(self$failures, list(
     list(self$file, context, test, result)
    ))
   self$cat_tight("\n   ", format(result), "\n")
  },
  end_test = function(context, test)
   self$cat_tight("\n"),
  end_context = function(context)
   self$cat_tight("\n"),
  end_file = function()
   self$cat_tight("\n"),
  end_reporter = function() {
   if (!length(self$failures)) return()
   self$cat_tight("\nFailures:\n")
   for (f in self$failures) self$cat_tight(
    " File: ", f[[1]],
    " Context: ", f[[2]],
    " Test: ", f[[3]],
    " Result: ", format(f[[4]]),
    "\n"
   )
  }

Hi everyone,

I have some wondering about the restoration of the user's par() in a package's functions. There are several discussions on this list and elsewhere about the general problem, but I haven't found any advice regarding the following specific point. Very thankful to anyone who can give me insight.

Let's say your package has a function that produces a plot by calling core-R functions like graphics::matplot(). These core-R functions are called with specific parameters, owing to the nature of the plots and of what needs to be emphasized. Here's a silly example:

```
pkgfun <- function(){
   graphics::matplot(x = 1:2, y = 1:2, type = 'l')
}
```

The function above creates a plot on some graphic device and, *while the device is open*, the user's par() is actually modified. For example we have

original par():
$xaxp
[1] 0 1 5

modified par() while graphic device is open:
$xaxp
[1] 1 2 5

This modification occurs simply because we have called matplot(), even if we didn't directly call par(... = ...). Note that the original par is restored once we close the graphics device, say with dev.off() or closing its window.

Here I have a first question:

? Does CRAN mean that the user's par() should be restored *once the graphics device is closed*? or right *when the function has finished executing*, even if the graphics device is still open?

I wasn't unable to understand which is meant, from the documentation.

If the answer is "only once the graphics device is closed", then the function pkgfun() is OK and I don't need to worry further (and you don't need to read further).


Let's now assume that CRAN wants the par() restored immediately after the function finishes executing instead. Then pkgfun() above is *not* acceptable in a package (destined to CRAN). One should save the old par() and restore it with on.exit(), as follows:

```
pkgfun2 <- function(){
   oldpar <- par(no.readonly = TRUE)
   on.exit(par(oldpar))

   graphics::matplot(x = 1:2, y = 1:2, type = 'l')
}
```

This new function version restores the original par() right after its execution.

However, I now see a problem with this. Imagine that the user wants to add some annotation or similar to the plot. Let's say they want to add a vertical red line at x?=?1.2. With the original function, the user could simply achieve this with

```
> pkgfun() # produces the plot
> abline(v = 1.2, col = 'red') # user wants to emphasize this x-value
```

then saving the plot somehow, maybe sandwiching the two commands above between a pdf() and a dev.off().

With the "correct" function pkgfun2(), however, this is not possible. No vertical red line will appear, because the graphics parameters in par() no longer reflect that particular graphics window. How to deal with such a case?

Sorry for the long email, hope it's of interest to others!
Kind regards,
Luca

Luca,

I think you misunderstand the rule: the purpose is to not modify *your* customisation in par() that overrides *graphical* parameters (like line types, colors etc.), it is not about plot coordinate setup which you clearly are supposed to keep if the purpose of the function is to setup a new plot. Also, yes, you have to restore it at the end of the function.

To take your example (slightly expanded):

> a=par(no.readonly = TRUE)
> graphics::matplot(x = 1:2, y = 1:2, type = 'l', lwd=2, col=2)
> b=par(no.readonly = TRUE)
> all.equal(a,b)
[1] "Component ?usr?: Mean relative difference: 2"
[2] "Component ?xaxp?: Mean relative difference: 2"
[3] "Component ?yaxp?: Mean relative difference: 2"


The key here is that matplot makes sure that parameters it changed (like lwd or col) are then restored before the function exits, so the only remaining changes are in parameters that setup the coordinates, but nothing else.

This also means that your pkgfun doesn't need to mess with par() at all. The whole point of the rule is that, if everyone follows it, then you only need to restore par() if you modified it directly using par() - all other functions should restore things correctly.

In addition, you typically don't want to save everything - you only need to save what you changed, so the typical idiom you should use is more like:

f = function() {
 p = par(family="Times")
 on.exit(par(p))
 graphics::matplot(x = 1:2, y = 1:2, type = 'l')
}


> a=par(no.readonly = TRUE)
> f()
> b=par(no.readonly = TRUE)
> all.equal(a,b)
[1] "Component ?usr?: Mean relative difference: 2"  [2] "Component ?xaxp?: Mean relative difference: 2"
[3] "Component ?yaxp?: Mean relative difference: 2"

Cheers,
Simon

Simon, I agree that the three parameters below are fine to modify, but 
some that are more problematic are the outer margin and multiple plot 
settings, e.g. oma and mfrow.

Changing those will affect later plots, so they should be reset.  But if 
they are reset, it becomes difficult to add to the plots.  Perhaps the 
community docs should discuss this issue?

I think the best approach is to reset them, but return the required 
settings so a later function can temporarily restore them for long 
enough to modify the plot, i.e. the sort of thing persp() does.

Duncan Murdoch

One additional option for allowing the user of your function to add to
a plot, but still properly reset graphical parameters, is to include
an optional argument to your function that can be a function that will
be run inside of your function, before graphical parameters are reset.
One advantage of doing this is that you can have your function run the
user supplied function after the coordinate system is set up, but
before the main data is plotted, this means that the data will be "on
top" of the user additions instead of underneath.  The panel.first
argument to the hist function is an example of this (though it expects
an expression rather than a function to be passed in, the function
option is probably safer).

   I asked Claude for a fine-grained report on timing. This is on
Linux, so the total check time is only 471 seconds, and there's no
guarantee of perfect proportionality when moving from Linux to Windows
(i.e., some components could change their proportional contribution
significantly ...) -- but it does suggest that there's really not much

R CMD check --as-cran on oce_1.8-3
===================================

WHERE THE TIME WENT
--------------------------------------------------------------------
Phase                                              Elapsed    % total
--------------------------------------------------------------------
Installing package (compile C/Fortran,               94s       20.0%
  45 source files, 13 MB installed libs)
checking R code for possible problems                 63s       13.4%
  (codetools static analysis, 63 R files)
Examples, --run-donttest pass                          52s       11.0%
Examples, normal pass                                  46s        9.8%
HTML manual (592 man pages)                            43s        9.1%
PDF manual (Rd -> LaTeX -> PDF, 592 pages)              40s        8.5%
Tests (testthat, 3098 assertions)                       29s        6.2%
CRAN incoming feasibility (network check)               23s        4.9%
Re-building vignette outputs (7 .Rmd vignettes)         18s        3.8%
--------------------------------------------------------------------
Subtotal, top 9 phases                                 408s       86.6%
Everything else (~250 fast structural/                  63s       13.4%
  metadata/Rd-syntax checks)
--------------------------------------------------------------------
TOTAL                                                  471s      100.0%

DIAGNOSIS
---------
No single pathological bottleneck -- time is spread across genuinely
large-package overhead, not one runaway step.

- Compilation (94s) is the single biggest cost, driven by the compiled
  C/Fortran extensions (13 MB of installed libs/).
- Examples run twice (normal + --run-donttest, as --as-cran mandates):
  combined 98s (~21%), more than installation. Individual examples are
  all fast (none over 1.2s per oce-Ex.timings; per-example self-times
  sum to only 27.7s, so the rest is per-run R startup/package-load
  overhead x2).
- Manual generation (PDF+HTML, 83s combined, ~18%) is proportional to
  the 592 .Rd pages -- pure documentation volume, not a specific slow
  page.
- codetools (63s) scales with R source size/complexity; not unusual
  for 63 source files with S4 methods.
- Tests and vignette rebuilding are comparatively cheap (29s and 18s)
  despite 60 test files and 7 vignettes.

Bottom line: this is what a large, compiled, richly-documented package
check looks like -- cost is distributed roughly evenly across install/
examples/manual/codetools, with no outlier worth specifically
optimizing unless trimming compiled-code size or \donttest example
runtime.

My apologies.. I missed the grandfathered status.

That said, this package desperately exhibits the main downside of the OOP deep inheritance design philosophy... tight coupling of a large code base. If a more functional approach with OO composition instead of inheritance had been used then it would be relatively easy to split the package up for different use cases with only shared code package dependencies to couple them. Having been burned by the promises of inheritance early in my career I empathize with Dan's plight but would also point out the sunk-cost fallacy will only get worse with time... eventually you have to re-design or die of excess code linkage.

If the package could be split, the relevant documentation could also be split and users would not be saddled with loading code they won't be using. But splitting an S4 inheritance tree is trouble-prone and this migration will almost certainly break existing user code so it would probably need a switch away from S4 and it won't happen quickly. A new package name would probably be least confusing.
