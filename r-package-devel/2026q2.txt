Dear members,

I would like to highlight an ecosystem-wide issue regarding lower-level
(not in-build in R) packages significantly increasing their minimum
required R version in their Depends or Imports fields.

A recent (a few months) example is the gsl package, which was updated to
require Depends: R (>= 4.5.0). Because of this, all 200+ of its reverse
dependencies now implicitly inherit this strict requirement.

This creates a highly misleading experience for everyday users. For
instance, the copula package explicitly states Depends: R (>= 3.6.0) on
CRAN. However, in reality, it requires R >= 4.5.0 to successfully
install.packages.

A user can try to install an older version of a package but is that
expected user experience.
Moreover I know many R users who are not even aware that this is an option.

What is the current policy and CRAN expectations in such situations?

Thank you for your support.

Best regards,

Maciej Nasinski
University of Warsaw

I don't think there's a policy about this. But there's the paradox
that a package, like gsl with v4.5.0, may depend on R >= r-release,
and yet it's being tested and should therefore work on r-oldrel (4.4.0
in this case). So I think that such dependency should generate an
ERROR.

I also see that this is being discussed upstream in [1], so adding the
link for visibility.

[1] https://github.com/RobinHankin/gsl/issues/28

On 9 April 2026 at 16:24, Maciej Nasinski wrote:
| What is the current policy and CRAN expectations in such situations?

The only written 'policy' is, as you likely know, at

 https://cran.r-project.org/web/packages/policies.html

The recursive nature of dependencies certainly applies as you observe. I
would offer two comments based on my experience with R:

 i)  CRAN operates 'at @HEAD' (to use a software engineering term). It cares
     mostly about _current packages_ working flawlessly together on the
     _current R version_ (and effectively one version back).

     That fits within the realm of the constraint you saw imposed.

     Asking for more i.e. 'any combination of packages at any past moment in
     time' is not practical given the CRAN resources (of four people, give or
     take, for three decades -- their choice).

 ii) All this is still open source. You could patch your local copy of 'gsl'
     and remove the constraint, or use an older versions, or do something
     else creative.

     The power of _local_ repositories that _you_ control should not be
     underestimated. I helped a little to use this power via the `drat`
     package; these days it is even easier to set up an r-universe repo which
     will also give you binaries.

In short, it is what it is "for a reason" as well as history, but you are
free to navigate around it.

Hope this helps, Dirk

Part of the problem is that it's not easy to verify that the minimal
declared R version is actually correct, so in many cases it stays
incorrect. Besides, which one is "correct"? Is it the oldest one where
the package installs? Passes R CMD check? Which NOTES or WARNINGs are
acceptable?

Here are the minimal R versions that packages on CRAN currently ask for:

tools::CRAN_package_db() -> db
db$Depends |> lapply(\(x)
 if (!is.na(x)) {
  v <- tools:::.split_dependencies(x)[['R']]
  if (!is.null(v) && v$op == '>=')
   as.character(v$version)
 }
) |> unlist() |> table() -> versions
versions[versions |> names() |> numeric_version() |> order()] |> plot()

The three largest peaks are 3.5, 2.10, and 4.1, likely added
automatically by R CMD build. Usually it's an optimistic lower bound,
at worst the maintainer's requirement is overly strict (as you observed
with 'gsl'). Sometimes the R version in question (e.g. 3.8, 3.10, 3.10,
3.21, 3.50, 3.60, 3.63) never existed at all.

Unfortunately, the proj/ and gdal/ directories were not built into the macOS R-4.6.0 arm64 binary package of 'terra'. The proj/ and gdal/ directories are present in the macOS R-4.6.0 x86_64 'terra', and as far as I can see in 'sf' on macOS and Windows. The 'oce' Windows case may be transient, and would be worth checking locally if possible with R 4.6.0 and a binary install of 'sf' and other packages.

In 'terra' as in other packages inheriting configure.ac from defunct 'rgdal', the configure argument: --with-data-copy=yes is needed to ensure that the GDAL and PROJ metadata corresponding exactly to the GDAL and PROJ versions used for building the binary package is copied into the binary package. The proj.db SQLite file is also versioned, as new tables are added to proj.db.

Some time ago we did look at shipping a data only package with version(s) of proj.db, but then found that users may have installed 'terra' with an earlier PROJ and proj.db, and 'sf' with later ones, leading to confusion. The proj.db is about 10.4MB, up from 9.1MB two years ago, but growing as the table count and complexity grows especially for transformations between datums.

Andrea,

if the bug affects your use case then you should use 4.1.3 otherwise 4.1.0. Normally, ABI compatibility is guaranteed across all patch versions - hence the comment to make sure the requirement is not unnecessary, but if your package does not work with 4.1.0 but does with 4.1.3 then it's a perfectly justified special case.

Andrea,

At this distance in time it is mostly immaterial whether you depend on a patch release. On the other hand, the gain from not just depending on 4.2.0+ must also be negligible by now. 

The reason for the policy was an incident somewhere around R 3.3-3.4, when a relatively minor package (pbkrtest) decided to depend on a patch release (possibly due to an updated sigma() function in 3.4.2, but I may misremember). However, pbkrtest had quite a lot of reverse dependencies, including Rcmdr, which several sites were using for teaching. After that point sites that updated their installation yearly suddenly could not install Rcmdr on their auditorium/computer lab setups, whether or not they actually intenden to use pbkrtest for anything. For the same reason, we have become much more careful not to introduce new functionality in base R between major releases.

-pd
